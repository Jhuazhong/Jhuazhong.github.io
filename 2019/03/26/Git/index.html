Git本地版本控制

git命令跟linux命令兼容

本地库初始化

  • 命令git init
    • 在你希望作为本地库的目录下用该命令

注意：.git目录存放的是本地库相关的子目录和文件，不要删除，也不要随意修改

设置签名

  • 形式
    • 用户名：xxx
    • Email地址:xxx@xxx.com
  • 作用：区分不同开发人员身份
  • 注意：这里设置的签名和登陆远程库（代码托管中心）的账号、密码没有任何关系
  • 命令
    • 项目级别/仓库级别：仅在当前本地库范围内有效
      • git config user.name xxx
      • git config user.email xxx@xxx.com
      • 信息保存的位置：当前项目目录下.git/config文件
    • 系统用户级别：登陆当前操作系统的用户范围
      • git config --global user.name xxx
      • git config --global user.email xxx@xxx.com
      • 信息保存的位置：cat ~/.gitconfig
    • 级别优先级
      • 就近原则：项目级别优先于系统用户级别，二者都有时，采用项目级别的签名；如果只有系统用户级别的签名，就以系统用户级别的签名为准
      • 二者都没有是不允许的

添加提交及查看状态操作

git 命令都是Git专属的命令

  • git status查看工作区暂存区本地库的状态
  • git add 文件名.格式名将文件提交到暂存区
  • git rm --cached <文件>把暂存区的该文件移除，而该文件仍旧在工作区
    • git -rf <文件>强制删除
  • git commit <文件>把暂存区的内容提交到本地库

工作区修改已经追踪的文件

  • git status还是可以查看状态
  • 这时工作区以追踪的修改文件可以有两种操作
    • 用git add <文件>在暂存区更新文件修改
    • 也可以用git commit <文件>的方式直接把修改文件提交到本地库
    • git commit -m "commit message" <文件>可以不用调用外部编辑器直接写文件修改注释

版本前进&后退

查看历史记录

  • git log
    • 多屏显示控制方式：
      • 空格向下翻页
      • b向上翻页
      • q退出

如果修改的记录很多，可以用以下命令显示简略信息

  • git log --pretty=oneline
  • 或者git log --oneline
    • 在上一步的基础上省略了部分哈希值，更简洁

还有命令

git reflog可以显示移动到某个版本需要多少步

基于索引值前进&回退版本【推荐】

  • git reset --hard 局部索引值 这里索引值是reflog简写哈希值
    • 配合git reflog或者git log --oneline

使用^和~符号的版本选择

  • 这种方式只能回退不能前进
  • git reset --hard HEAD^
    • 几个^表示退几步，但这种方式对于回退次数太多时很麻烦
  • git reset --hard HEAD~n
    • 表示后退n步

reset命令的三个参数的对比

  • git reset --soft
    • 仅在本地库移动HEAD指针
  • git reset --mixed
    • 在本地库移动HEAD指针，重置暂存区
  • git reset --hard
    • 在本地库移动HEAD指针，重置暂存区，重置工作区

查看命令帮助

  • git help 命令
    • 例如git help reset

永久删除文件找回

前提：删除前，文件存在时的状态提交到了本地库

本地库中文件删除

  • rm <文件>删除文件
  • git add <文件> 将删除文件的操作添加到暂存区
  • git commit -m "delete 文件"将删除文件的操作添加到本地库

找回删除文件

  • 通过前面版本回退的方式找回删除的文件
    • git reflog
    • git reset --hard 局部索引值

比较文件

  • git diff <文件>
    • 工作区文件和跟暂存区对应文件比较
  • git diff <本地库中历史版本> <文件名>
    • 将工作区中文件和本地库中历史版本比较，例如HEAD对应版本
  • 如果不加文件则比较所有文件

分支管理

什么是分支

在版本控制中，使用多条线推进任务

各个分支在开发的过程中是彼此独立的，除非合并了分支。

分支概述

  • 主干master
  • 其他功能用分支开发，例如feature_game
  • 修改bug一般创建一个hot_fix（热修复，服务不停）分支

分支的好处

  • 同时并行推进多个功能开发，提高开发效率
  • 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响。
    • 失败的分支删除重新开始即可

分支操作

  • git branch -v可以查看分支数和当前所处分支
  • git branch 分支名创建一个新的分支
  • git checkout 分支名切换分支
  • 合并分支
    • 第一步：切换到接收修改的分支上
      • git check out 分支名
    • step2：git merge 待合并的分支
  • 解决冲突
    • 冲突的表现：在文件中标出冲突的部分
    • 解决方式：
      • 手动修改文件冲突内容，删除特殊符号
      • git add <文件名>
      • git commit -m "日志信息"
        • 注意：此时commit后不加文件名

Github—远程库

本地库与远程库的交互

本地和远程库绑定

  • git remote -v
    • 查看远程库的别名
  • gir remote add 别名[例如origin] [远程库的http地址]
  • 推送push
    • git push 别名[如origin] [本地分支]
  • 克隆clone，这个操作有三个效果
    • 第二人从第一人的远程库clone项目到第二人的本地
    • 创建origin远程地址别名
    • 初始化本地库
    • 如果第二人在本地库修改向提交到第一人的远程库
      • 这个时候执行git push origin [分支]会失败，没有权限
      • 需要先由第一人把第二人加入到团队

远程库修改的拉取pull

第一人拉取pull远程库的修改

  • pull = ferch+merge
    • git fetch [远程库别名] [远程分支名]
      • fetch操作把远程库下载到本地，但是不修改本地库中内容
        • 可以在远程库别名/远程分支名中查看，用命令git checkout 远程库别名/远程分支名
    • git merge [远程库别名/远程分支名]把远程库下载到本地的内容合并到本地库
  • git pull [远程库别名] [远程分支名]
    • 如果是异行修改，就直接用pull
    • 如果是同行修改，分开用fetch+merge

协同开发时冲突的解决

  • 如果不是基于Github远程库的最新版所作的修改，不能推送push，必须先拉取pull
  • 拉取下来后如果进入冲突状态，则按照“分支冲突解决”操作即可

跨团队协作

  • 第三个人（不在团队中）通过fork的方式复制一个“第一人的远程库”
    • 这个远程库属于第三人
    • 通过git clone [第三人fork的远程库地址]的方式得到本地库
    • 把修改的内容push到远程库
  • 接着在第三人的远程库pull request---->new pull request
  • 第一人点pull request看到修改的内容，审核内容
  • 双方可以通过Github的pull request的板块互动
  • 如果没问题就点击merge pull request
  • 将远程库修改pull到本地

SSH免密登陆

因为通过HTTP方式push代码到远程库每次都需要用户名登陆密码，这里介绍另一种SSH方法。

  • 进入当前用户的家目录

    • cd ~

  • 删除.ssh目录

    • rm -rvf .ssh

  • 运行命令生成.ssh密钥目录

    • ssh-keygen -t rsa -C [Girhub邮箱账号]

  • 进入.ssh目录查看文件列表

    • cd .ssh
    • ls -IF

  • 查看id_rsa.pub文件内容

    • cat id_rsa.pub

  • 复制id_rsa.pub内容，登陆Github.点击用户头像—>Settings—->SSH and GPG keys

  • New SSH Key

  • 复制密钥信息

    • 测试是否成功

  • 回到Git Bash创建远程地址别名

    • git remote [别名，如xxx_ssh] [SSH地址]

  • 然后就可以用推送了

    • git push [远程地址别名] [分支名]
